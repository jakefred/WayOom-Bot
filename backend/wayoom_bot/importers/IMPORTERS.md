# Importers

This package converts third-party flashcard formats into WayOom model kwargs. The caller (the view layer) is responsible for saving the returned objects inside a database transaction.

---

## apkg.py — Anki .apkg importer

### Public API

```python
parse_apkg(file_bytes: bytes) -> ParseResult
```

Accepts raw bytes of an `.apkg` file and returns a `ParseResult` containing lists of deck kwargs, card kwargs, media entries, and error strings. No database access happens inside the parser — the view resolves FKs and persists objects.

### Supported Anki formats

| Archive file | Schema | Metadata source | Notes |
|---|---|---|---|
| `collection.anki2` | v11 | `col` table (`models`/`decks` JSON columns) | Anki 2.0 |
| `collection.anki21` | v11 | `col` table (same as above) | Anki 2.1 |
| `collection.anki21b` | v18 | `notetypes`/`decks`/`fields`/`templates` tables | Anki 2.1.50+, zstd-compressed SQLite, Protobuf template configs |

Format detection is automatic — `_detect_format()` checks which collection file exists in the zip archive.

### Key design decisions

- **Deterministic UUIDs.** Card and deck IDs are UUID v5 derived from Anki source identifiers (`note_guid:card_ord` for cards, `deck:{anki_id}:user:{user_id}` for decks). This makes re-imports idempotent — `bulk_create(ignore_conflicts=True)` silently skips duplicates.

- **`_WAYOOM_NAMESPACE` must never change.** It is the UUID v5 namespace for all generated IDs. Changing it would invalidate every existing dedup key and cause all cards to be re-imported as new records.

- **`text_factory = bytes` on the SQLite connection.** Anki databases from older versions or non-English locales may store TEXT columns in Latin-1 or Windows-1252 rather than UTF-8. Python's `sqlite3` module would raise `UnicodeDecodeError` with its default `str` text factory. All text columns are decoded through `_decode_text()` which uses a UTF-8 → Latin-1 fallback chain.

- **Parser never touches the database.** It returns plain dicts with model field kwargs. The view (`ApkgImportView`) handles `get_or_create`, `bulk_create`, user scoping, and media file storage inside a transaction.

- **Media is collection-scoped.** Anki stores media at the collection level (any card can reference any file). The parser extracts all media into `ParseResult.media`. The view associates each file with every deck in the import.

### Data flow

```
.apkg (zip) → _detect_format() → raw SQLite bytes
    → _parse_col_meta_v11() or _parse_col_meta_v18() → models + decks metadata
    → _parse_collection() joins notes + cards → _build_card_kwargs() per row
    → _extract_media() reads the zip media manifest and file bytes
    → ParseResult(decks, cards, media, errors)
```

### Common pitfalls

- **Protobuf template extraction is best-effort.** In v18, template `qfmt`/`afmt` are inside Protobuf config blobs. `_extract_template_qfmt()` does a manual scan for `{{`-containing strings rather than using a full Protobuf parser. Cloze detection falls back to checking the notetype name if this extraction fails.

- **Anki's `due` field meaning depends on `queue`, not `type`.** See `_convert_due_date()` — review queue means days-since-creation, learning queue means absolute Unix timestamp, new queue means position (not a date).

- **The `extra_notes` list is capped at 20 fields** (`_MAX_EXTRA_FIELDS`) to prevent excessively wide Anki note types from bloating the database.

### Tests

Parser tests live in `wayoom_bot/tests.py` under `ApkgParserTests`. Test fixtures are pre-built `.apkg` files in `test_fixtures/`, generated by `test_fixtures/build_fixtures.py`. The `test_non_utf8_fields_parsed_without_error` test specifically covers the Latin-1 encoding fallback.
